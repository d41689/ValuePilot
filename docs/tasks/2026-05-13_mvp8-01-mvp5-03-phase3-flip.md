## Operator Runbooks

### Re-enriching `review_needed:*` CUSIP mappings

The OpenFIGI auto-confirm path skips CUSIPs whose response had
conflicting tickers or no US Common Stock match
(`evaluate_openfigi_matches` returns `confidence='review_needed:*'`).
Those mappings stay in `cusip_ticker_map` with the `review_needed`
confidence. The standard `enrich_unmapped_holdings` worker only
processes `cusip_mapping_status='pending_mapping'` and will not
re-fetch already-mapped CUSIPs.

To re-resolve a specific batch (e.g. after OpenFIGI improves
coverage for foreign-listed CUSIPs):

```sql
-- 1. Identify the CUSIPs you want to re-resolve.
SELECT cusip, ticker, confidence, mapping_reason
FROM cusip_ticker_map
WHERE is_active = true
  AND confidence LIKE 'review_needed:%'
ORDER BY cusip;

-- 2. Deactivate the existing mappings.
UPDATE cusip_ticker_map
SET is_active = false
WHERE cusip IN (...)
  AND is_active = true
  AND confidence LIKE 'review_needed:%';

-- 3. Flip the dependent holdings back to pending so the
--    enrichment worker re-fetches them.
UPDATE holdings_13f
SET cusip_mapping_status = 'pending_mapping', stock_id = NULL
WHERE cusip IN (...)
  AND cusip_mapping_status = 'linked';
```

Then re-run `enrich_cusips_from_openfigi(db)` until convergence
and `backfill_stock_ids(db)`.

Alternatively, for hand-curated ADR/PLC tickers (TSMC, Aon PLC,
Flutter, Diageo, HDFC, Medtronic, Accenture, etc.), insert a
manual row with `source='manual', confidence='manual'` — the
`upsert_cusip_mapping` confidence ranking will keep it ahead of
any future OpenFIGI re-resolution attempt.

A dedicated re-enrichment endpoint / admin button is a follow-up
ticket (Staff Eng tech-debt list); not blocking MVP8-01.
