## Q1 — `is_current` Option A long-term durability

The decision is correct **today**. Multiple `is_current=True` rows per `(stock_id, metric_key)` for fiscal time series is exactly what the Piotroski / screener / formula stack needs. Wiping it would be a Sev-1 product regression dressed up as a schema cleanup.

**Concerns:**

1. **The trigger condition for reopening is too soft.** The contract in `AGENTS.md` says "reopen the gate if a second opinion-metric consumer cannot use the tiebreak." In practice, the gating happens at code-review time, not at design time. A new consumer is more likely to ship a naive `WHERE is_current=True` query, pass review (the reviewer doesn't know about the opinion/fiscal distinction unless they read AGENTS.md end-to-end), and surface stale opinions to users. **Recommendation**: add a runtime guard or test fixture that exposes a stock with two `is_current=True` opinion rows and asserts that any new endpoint touching opinion metrics returns the most recent one. This makes the trigger condition mechanical rather than tribal.

2. **Alerting / notification surfaces are the canonical "second consumer."** When the project ships a "price target changed" alert, the natural query is `WHERE is_current=True` — that's the bug class Option B exists to prevent. No alerting is shipped today, so this is future backlog, but flag it now so the alerting ticket reopens the gate before code.

3. **The UI label `(VL report dated YYYY-MM-DD)` is precise and SME-acceptable.** Reads cleanly as "this is the date the analyst published, not a fiscal period."

## Q2 — Watchlist drawer M3 coverage limitations

The "Value Line data is not available for this stock in the current dataset" copy (post-review hardening; the original "not yet available" implied future ingestion) is **honest and shippable**.

**Concerns:**

1. **The asymmetry between 13F columns (universal) and M3 panel (5/240 stocks) is a real UX risk.** A user scanning the watchlist sees 13F signals on every row and then opens a drawer for a stock without VL coverage — the empty M3 panel reads as "ValuePilot doesn't know quality/valuation for this stock." That implication is correct, but the copy could be more useful. Suggested: change the empty state to two-line copy:
   ```
   Value Line data is not available for this stock in the current dataset.
   Quality & valuation overlay coverage is currently limited to a curated
   set of large- and mid-cap U.S. equities.
   ```
   This positions the gap as a coverage decision, not a per-stock failure.

2. **Missing signals investors actually use:**
   - **Dividend yield** — VL publishes this; not in the M3 panel. Income-focused screeners will miss it.
   - **Payout ratio** — same.
   - **Current price vs VL 18-month target as a single ratio** (e.g., "Current $X vs target $Y = -12%") — the user has to do the math from `vl_target_mid` + the watchlist's `price` column. The legacy Oracle's Lens page already computes `discount_to_reference` in `_valuation_payload`; the drawer doesn't surface it.
   - **Debt coverage / interest coverage** — leverage signals are in `metric_facts` (`leverage.long_term_debt_to_capital` is part of `QUALITY_METRIC_KEYS` in `dashboard.py`) but not in `_M3_METRIC_KEYS` in the watchlist endpoint. The two surfaces show different quality cuts.

   This is **not a block** — V1 scope was "compact panel." But the cross-surface inconsistency is the kind of thing investors do notice. File as N2-adjacent (VL coverage track) or its own follow-on.

3. **`piotroski_status` open vocabulary** — `partial` vs `calculated` vs future `complete` is fine, but the drawer only labels `partial` (`*` suffix + footnote). A future `calculated` row will render as just "Piotroski X/Y" with no qualifier — fine if it's the full 9-indicator score, confusing if it's a different-meaning legacy variant. The frontend should defensively render an unknown-status footnote, or the backend should normalize the vocabulary.

## Q3 — The four 13F columns on /watchlist

Conviction percentile / Δ Holders / Distinctiveness / Caveats is the **right V1 cut**. Each column maps to a question an investor actually asks:

- *"Is this name agreed on by good managers?"* — conviction percentile.
- *"Is the position growing or shrinking this quarter?"* — Δ Holders.
- *"Is this a crowded mega-cap or an under-followed name?"* — Distinctiveness.
- *"Are there reasons to slow down?"* — Caveats.

**Concerns:**

1. **The single column I would ADD: "Top holder weight" or "Best-holder confidence."** What's missing from the four-column set is "the *quality* of the holders" — Conviction percentile rolls quality into a ranking but loses the underlying signal. For a stock with conviction=70th percentile, an investor wants to know whether that comes from one Buffett 8% position or five 0.5% positions. The data is in the drawer (top_holders[:3] with `position_weight`), but the row needs a hint. A "Top holder X%" chip or a "1 high-conviction holder" badge would close that gap without adding a heavy column.

2. **Caveats default sort = `desc` (worst-first) diverges from the Pre-MVP7-01 D1 spec.** The MVP7-06 task file documents this and the rationale ("users clicking the Caveats column usually want to see RISKY signals first"). **I agree with the divergence.** A user clicking the Caveats header is investigating risk, not celebrating clean rows. The default also matches the visual cue (warning icon at the top of the sorted list).

   The one risk: a user scanning the watchlist who isn't aware of the click semantics could see clean rows hidden below caveat-heavy rows and walk away with the wrong impression. But the watchlist's default sort is MOS desc, not Caveats — Caveats sort only fires on explicit click — so this is unlikely.

3. **Score confidence is in the payload but not on the row.** `score_confidence: 'high' | 'medium' | 'low'` is part of the response shape; it shows up implicitly through the Caveat severity but is otherwise hidden until the drawer. For a research user, "this signal has low confidence" is a more decision-critical fact than "this signal has 1 caveat flag." Consider surfacing in V1.1.

## Q4 — Kahn Brothers True Positive caveat

The MVP8-03A admin amber banner is a **correct V1 patch** for a known data anomaly. But hardcoding the filer CIK is fragile.

**Recommendation (not blocking):**

Add a `manager_value_unit` column or `manager_reporting_overrides` table so the system can flag any filer whose `value_thousands` is actually in dollars. The Kahn Brothers banner becomes a row in that table, not a code-level constant. Benefits:
- Adding a second non-standard filer becomes a SQL insert, not a deploy.
- The admin reconciliation surface can show "this filer is on the override list and reports in dollars — warnings are expected" inline, not just in a banner that fires on quarter selection.

**Other filers in my domain knowledge that historically have had non-standard reporting:**
- **Berkshire Hathaway's confidential treatment** (already handled via `Filing13F.has_confidential_treatment` → `CONFIDENTIAL_TREATMENT_CAVEAT`). Periodic; not a unit issue but a similar "this filer needs special handling" pattern.
- **Renaissance Technologies and other quant filers** sometimes file co-attributed holdings (`otherManagers` populated) at scale, which the codebase handles via the `holding_attribution_status` filter. Worth confirming end-to-end if Renaissance is in the universe.
- **No other filers I know of** with the Kahn Brothers `value_thousands = dollars` pattern. But the V1 framework should assume one will appear.

## Q5 — Manager type taxonomy + admin classification (MVP8-03B B1)

The dual-chip display (`Derived: X` / `Admin: Y`) is the **right pattern for an operator/admin user**. It surfaces the divergence between what behavior says and what curation says — important for the SME refining the classifier.

**Concern for the RESEARCH user (the target persona for /watchlist):**

A research user does not need to see the divergence chip-by-chip. They want one answer: "what kind of manager is this?" The current dual-chip implementation:

- Suppresses when `admin == "unknown"` (correct — uncurated admin shouldn't pollute the UI).
- Shows two chips when both are non-unknown AND they differ.
- Shows one chip when they agree.

For an SME / admin reviewer this is honest. For a research user, two chips on the same manager row reads as confusing — "is this manager LTF or quant?" The intent is "behavior says LTF, admin says quant; we trust admin," but that nuance isn't legible from `Derived: LTF` / `Admin: Quant`.

**Recommendation**:

In the **research drawer** (Watchlist13FDrawer.tsx as-is), keep dual chips but add a tooltip on each chip that explicitly says which one is canonical. The current `title` attributes say "Behavior-derived manager type (overrides admin classification)" — that's contradictory. Either:

- (a) Behavior derived is canonical and admin is documentary → tooltip on the Admin chip should say "Admin curation, currently overridden by behavior derivation."
- (b) Admin classified is canonical and derived is documentary → flip the precedence.

The code (in `_apply_manager_signal_profiles`) does (a): derived overrides admin. The UI copy should match.

**Alternative**: for the research drawer, fold the dual into a single chip with a tooltip:
```
[Long term fundamental ⓘ]   ← single chip
              ↓ hover
              Behavior-derived. Admin classified this manager
              as "Quant"; behavior derivation took precedence.
```

This keeps the research user's mental model clean and still surfaces the divergence to anyone who hovers.

## Q6 — Cross-MVP signal coherence

The two surfaces (`/13f/oracles-lens` and the Watchlist drawer) should tell the same story for the 5 overlap stocks (ADBE, FICO, FNV, GOOG, MTDR in dev). Per the D2 unification, both call `_m3_facts_by_stock` and use `_fact_value` for the `value_numeric`-or-`value_json['partial_score']` fallback.

**Items I verified by reading the code:**

- **Piotroski score** — both surfaces extract from the same `MetricFact.value_json['partial_score']` via `_fact_value` / `_m3_panel_for_stock`. They will agree.
- **Earnings predictability** — same source (`quality.earnings_predictability.value_numeric`). They will agree.
- **VL target midpoint** — legacy page reads it via `_valuation_payload` (writes to `valuation_reference`); drawer reads it via `_m3_panel_for_stock` (writes to `vl_target_mid`). Same row, same value, different field name in the response. They will agree numerically.
- **Caveat flags** — legacy and drawer both come from `_stock_payload.caution_flags` in the dashboard service. Same list.

**Items where the two surfaces DO tell different stories:**

1. **Legacy page shows `discount_to_reference` (current price vs VL target).** Drawer does not. A research user clicks from `/13f/oracles-lens` to `/watchlist` and the discount disappears. → Same data, missing in drawer. **Filed as Q2 #2 above.**

2. **Legacy page's `quality_overlay` includes `return_on_total_capital`, `return_on_equity`, `net_profit_margin`, `debt_to_capital`, `owner_earnings_yield` (six metrics).** Drawer's `quality_overlay` shows Piotroski + earnings predictability only (two signals). → The drawer is a *compact* panel by design (MVP8-A2 scope), but the quality cut is markedly thinner than the legacy page. This is intentional but creates the impression that "drawer M3 < page M3."

3. **Score confidence vocabulary normalization** — legacy page uses `OWNERSHIP_SIGNAL_CONFIDENCE_LEVELS` ("high_confidence" etc.); watchlist API normalizes to "high"/"medium"/"low" at the boundary. A SME comparing the two surfaces sees "high_confidence" on one and "high" on the other for the same stock. Functionally consistent; cosmetically two-faced.

**Net**: The two surfaces are *factually* consistent (same DB rows, same scoring). They are *visually* different — drawer is a tighter cut of the same data. For V1, this is acceptable. For V1.1, a 3-line "View on Oracle's Lens dashboard for deeper quality cut" link in the drawer would close the loop.
