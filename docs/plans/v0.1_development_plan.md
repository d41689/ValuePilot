# ValuePilot v0.1 Development Plan

This document outlines the development plan for ValuePilot v0.1, based on the `value-pilot-prd-v0.1.md`. The goal is to build a robust system for parsing, normalizing, and screening financial data from Value Line equity report PDFs.

## Phase 1: Foundation & Infrastructure
**Goal:** Establish the core system architecture, database schema, and project structure.

### 1.1 Project Setup & Configuration
- [ ] Initialize repository structure (Monorepo with `backend/` and `frontend/`).
- [ ] Configure Docker Compose for full-stack environment (FastAPI, Next.js, PostgreSQL).
- [ ] Set up linting, formatting (Ruff, Prettier), and type checking (MyPy, TypeScript).
- [ ] Configure CI/CD pipelines for automated testing and linting.

### 1.2 Database Implementation
- [ ] Implement SQLAlchemy models based on PRD Section C (Core Tables).
    - `users`, `stocks`
    - `pdf_documents`, `document_pages`, `parser_templates`
    - `metric_extractions`, `metric_facts`
    - `formulas`, `calculated_runs`
    - `stock_pools`, `pool_memberships`, `screening_rules`
    - `stock_prices`, `price_alerts`
    - `notification_settings`, `notification_events`
- [ ] Set up Alembic for database migrations.
- [ ] Verify schema relationships and constraints.

### 1.3 Core Backend Services
- [ ] specific authentication/authorization (User model integration).
- [ ] Basic API endpoints for system health and configuration.

---

## Phase 2: Ingestion & Parsing (The Core)
**Goal:** Enable robust ingestion of Value Line PDFs, text extraction, and metric parsing.

### 2.1 PDF Ingestion Pipeline
- [ ] Implement file upload API (`POST /api/v1/documents/upload`).
- [ ] Create `pdf_documents` records upon upload.
- [ ] Implement PDF text extraction (Native text layer first, fallback structure for OCR).
- [ ] Store raw text and original PDF (MinIO or local storage for dev).

### 2.2 Stock Identity Resolution
- [ ] Implement logic to extract `ticker` and `exchange` from PDF.
- [ ] Implement "Get or Create" logic for `stocks` table.
- [ ] Implement company name similarity check.
- [ ] Flag `identity_needs_review` if confidence is low.

### 2.3 Value Line Parser (V1 Template)
- [ ] Implement `value_line_equity_report_v1` parser logic.
- [ ] Extract Header/Ratings (Price, P/E, Div Yield, Ranks).
- [ ] Extract Target Price Ranges (18-month, Long-term).
- [ ] Extract Time-Series Tables (Sales, EPS, Div).
- [ ] Extract Financial Snapshot & Narrative.
- [ ] **Deliverable:** Populate `metric_extractions` with `raw_value_text` and `original_text_snippet`.

### 2.4 Normalization Layer
- [ ] Implement Scale Detection (k, mil, bil, etc.) per PRD Appendix B.3.
- [ ] Implement Unit Normalization (USD, USD/share, Ratio) per PRD Appendix B.2.
- [ ] Implement `metric_facts` population logic.
    - Transform `metric_extractions` -> `metric_facts`.
    - Handle `is_current` logic.

---

## Phase 3: Data Management & Calibration
**Goal:** Provide APIs and tools for users to review and correct parsed data.

### 3.1 Data Retrieval API
- [ ] `GET /api/v1/stocks/{stock_id}` (Overview).
- [ ] `GET /api/v1/documents/{document_id}/extractions` (Traceability view).
- [ ] `GET /api/v1/stocks/{stock_id}/facts` (Normalized data).

### 3.2 Correction Workflow
- [ ] `POST /api/v1/extractions/{extraction_id}/correct` endpoint.
    - Create new `metric_facts` entry with `source_type=manual`.
    - Update `metric_extractions` to mark `corrected_by_user`.
    - Trigger formula recalculation (hook for Phase 4).

### 3.3 Admin/Calibration UI (Backend Support)
- [ ] Support bounding box (`bbox_json`) storage and retrieval.
- [ ] API to serve PDF pages/images for the frontend viewer.

---

## Phase 4: Logic Engine (Formulas & Screening)
**Goal:** Enable dynamic calculation of metrics and powerful screening capabilities.

### 4.1 Formula Engine
- [ ] Implement Formula model and dependency graph resolution.
- [ ] Create expression parser/evaluator (safe subset of Python or custom DSL).
- [ ] Implement dependency tracking (`dependencies_json`).
- [ ] Implement Recalculation Trigger logic (Observer pattern on `metric_facts` updates).
- [ ] Store results in `calculated_runs` and authoritative outputs in `metric_facts`.

### 4.2 Screener Engine
- [ ] Implement `screening_rules` storage and validation.
- [ ] Build query compiler: JSON Rule -> SQLAlchemy/SQL.
- [ ] Ensure queries run against `metric_facts.value_numeric` for performance.
- [ ] `POST /api/v1/screener/run` endpoint.
